## Context

现有烧香体验由 `BurnSession` 状态机与 `SmokeScene` 粒子系统驱动，已实现点燃、燃烧、完成与通知。新增需求跨越传感器输入（CoreMotion）、渲染层（SpriteKit/SwiftUI）与系统状态呈现（ActivityKit），属于跨模块扩展。

## Goals / Non-Goals

**Goals:**
- 在支持重力传感器的设备上实现“倾斜即影响烟雾走向”的实时联动。
- 增加香灰随时间形成勾炉香并在燃尽后掉落的可视化过程。
- 点香后在支持设备通过灵动岛/锁屏持续展示燃烧状态与剩余时间。
- 提供平台降级策略，保证 iPad/macOS 或不支持灵动岛机型不出现功能断层。

**Non-Goals:**
- 不改动现有点燃顺序、燃烧时长（21 分钟）与完成语义。
- 不引入云端同步、账号体系或跨设备状态共享。
- 不追求真实物理级香灰堆积模拟（优先可控的演化动画表达）。

## Decisions

### 1. 烟雾联动采用“姿态向量 -> 受力参数”映射
- 方案：用 CoreMotion 输出 roll/pitch，经低通滤波后映射到粒子横向加速度与发射角偏移。
- 原因：不改写粒子系统架构即可获得稳定、可调的倾斜反馈。
- 备选：直接改写每帧粒子位置。未选，成本高且难以保持自然感。

### 2. 香灰采用“轨迹显现 + 终态掉落”两阶段
- 方案：燃烧阶段按 progress 渐进显示香灰轨迹；到 complete 时触发整体掉落动画并归于香炉。
- 原因：兼顾仪式感与可实现性，避免实时物理堆积带来的复杂度与性能波动。
- 备选：全粒子物理堆积。未选，跨平台稳定性和调参成本过高。

### 3. 系统状态展示采用 Live Activity，灵动岛为可见层
- 方案：在点香成功后启动 Live Activity，内容包含阶段与剩余时间；在不支持灵动岛设备自动降级到锁屏展示，必要时应用内补充状态条。
- 原因：灵动岛本质是 Live Activity 的一种展示面，统一数据源可减少分叉。
- 备选：仅本地通知。未选，无法满足“过程可见”的实时状态诉求。

### 4. 状态发布统一由 BurnSession 驱动
- 方案：保持 `BurnSession` 为唯一业务真源，新增发布器订阅 state/progress 推送到烟雾联动与 Live Activity。
- 原因：避免双向耦合与状态不一致，降低回归风险。

## Risks / Trade-offs

- [传感器抖动导致烟雾跳变] -> 加低通滤波与阈值钳制，限制最大偏移角。
- [Live Activity 平台碎片] -> 能力检测 + 降级路径（锁屏/应用内状态）统一处理。
- [香灰动画与现有烟雾层叠冲突] -> 分层渲染并固定 z-order，增加 snapshot 回归。
- [电量消耗上升] -> 降低传感器采样频率并在后台/非燃烧状态停用姿态更新。

## Migration Plan

- 增量发布，无数据迁移。
- 先落地 V1：重力烟雾 + 香灰轨迹 + Live Activity 基础文案。
- 若出现性能回归，可独立关闭倾斜联动或香灰动画开关，不影响主流程。

## Open Questions

- 香灰轨迹模板是否固定单一“勾炉香”还是支持多模板切换。
- Live Activity 是否需要交互按钮（如“打开应用”）还是纯展示。
- 低电量模式下是否自动降级传感器刷新率。
