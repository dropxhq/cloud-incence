## Context

当前实现与既有规格在三个点存在偏差：
1. 输入态退出后，idle 未保留展示已输入心愿。
2. 空心愿在 complete 态仍有固定等待，未直接进入祝语。
3. 粒子烟雾与香棒几何常量分散，存在错位风险。

该变更仅涉及本地 UI 与状态流，不引入新依赖、不改变持久化或网络层。

## Goals / Non-Goals

**Goals:**
- 让输入流程与规格一致：退出输入后保留并展示已有心愿文本。
- 让完成态分支与规格一致：空心愿直接进入祝语淡入。
- 统一香棒/烟雾几何参数来源，消除视觉错位。
- 保持既有状态机与主流程（idle -> composing -> lighting -> burning -> complete）不变。

**Non-Goals:**
- 不改动 21 分钟计时规则与通知策略。
- 不新增功能（历史心愿、持久化、社交）。
- 不重写动画框架（继续沿用 SwiftUI + SpriteKit）。

## Decisions

### 1) Idle 展示策略改为“有值显示、无值占位”
- 决策：在 idle 状态按 `prayerText.isEmpty` 分支渲染。
- 原因：满足“收起键盘后文字保留显示”的规格，同时避免用户误判输入丢失。
- 备选方案：始终显示占位按钮并在下方额外展示文本；未采纳，因为信息密度高且视觉噪音更大。

### 2) Complete 动画改为双分支时序
- 决策：
  - 空心愿：立即显示祝语并执行淡入。
  - 非空心愿：保留 2s 消散，再显示祝语。
- 原因：与规格对齐，并消除空心愿场景的“黑屏等待”。
- 备选方案：统一都走 2s 消散；未采纳，因为空内容场景缺少可感知动画对象。

### 3) 几何常量单一来源
- 决策：将烟雾发射点定位改为使用与香棒一致的布局常量（同值或注入），禁止双份魔法数漂移。
- 原因：这是错位问题根源，统一来源可从机制上避免回归。
- 备选方案：仅手工把 `37` 改成 `42` 或反向修改；未采纳，因为后续仍可能再次分叉。

### 4) 输入退出时显式收起焦点
- 决策：点击输入区外退出 composing 时，同时触发焦点清理。
- 原因：不同平台键盘行为存在差异，显式焦点收起可提升一致性。

## Risks / Trade-offs

- [动画节奏变化引发主观感知差异] -> 通过对比录屏在 iOS/macOS 双端回归验证。
- [常量统一改动影响已有视觉调参] -> 使用同一组快照对比，必要时微调单一常量源。
- [焦点收起实现方式耦合 View 结构] -> 采用最小侵入方式，仅在 composing->idle 切换时触发。

## Migration Plan

- 本变更为纯前端行为修复，无数据迁移。
- 发布策略：一次性随常规版本发布。
- 回滚策略：若出现严重视觉回归，可回滚到上一个 UI 提交，不影响数据或通知逻辑。

## Open Questions

- 是否需要在 idle 保留文本状态下提供“编辑”快捷入口（例如轻点文本直接回到 composing）？当前不作为本次必做项。
